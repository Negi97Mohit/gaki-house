1. Project Overview
   App Name: GAKI がき - House for Video Creation

Purpose: An AI-powered web application for creating, recording, and editing professional-looking videos. It integrates real-time captions, AI-generated HTML overlays, multi-scene management, and a post-production editor.

Key Features:

Live Studio: A multi-scene environment for composing video, screen shares, and overlays.

Session Recording: Records the entire canvas stream and all state changes (overlays, captions, layout) into a session file.

Video Editor: A post-production editor to review and export recorded sessions.

AI Overlays: Voice and text-prompt-based generation and updating of HTML/CSS/JS overlays via Gemini.

Real-time Captions: Live speech-to-text from Deepgram with dynamic, word-by-word animations.

Draggable Elements: Users can add, drag, and resize text, browsers, files (images, videos, PDFs), charts, and effects on the canvas.

Asset Library: Built-in search for stock images and GIFs.

Effects: CSS video filters, ambient particle effects, and real-time "Neon Edge" effect.

Scene Transitions: Customizable transitions (Dissolve, Slide, etc.) between scenes.

2. Architecture Summary
   Overall Architecture: This is a component-based application built with React. It follows a "smart page, dumb component" pattern, where the main page components (Index.tsx, EditPage.tsx) manage the majority of the application logic and state, passing props and callbacks down to a large tree of presentation and UI components.

State Management Approach:

Core App State: The primary state, scenes (an array of SceneState objects), is managed within the Index.tsx component using useState. This object tree contains everything about all scenes (overlays, settings, etc.). This is a form of centralized component state.

Global State: react-query is used for client-side data fetching (likely for assets). React Context is used for cross-cutting concerns: ThemeProvider (dark/light mode), LogProvider (logging), and DebugProvider (AI debugging).

Persistent State: localStorage is used to store gaki-recorded-sessions and gaki-saved-overlays, managed via the useLocalStorage hook.

Routing Strategy: Client-side routing is handled by react-router-dom. The application has three main routes defined in App.tsx:

/ -> Index: The main live studio/recording interface.

/edit/:sessionId -> EditPage: The post-recording video editor.

- -> NotFound: A 404 page.

Data Flow (Text Diagram):

AI Command (Voice): Mic -> useContinuousAudio -> useDeepgramSpeech -> VideoCanvas (sets transcript state) -> handleFinalTranscript -> onProcessTranscript -> ai.ts (processCommandWithAgent) -> Gemini API -> ai.ts (returns HTML) -> Index.tsx (updateActiveScene) -> VideoCanvas (renders DraggableOverlay).

AI Command (Text): AICommandPopover (submit) -> onProcessTranscript -> (Same as above).

Session Recording: User Interaction (e.g., drag overlay) -> Draggable... component callback -> Index.tsx (updates scenes state) -> Index.tsx callback calls recording.record...() -> useRecordingSession (pushes keyframe to ref array). Stop Record Click -> Index.tsx -> recording.stopRecording() -> useRecordingSession (creates Blob, bundles tracks) -> Index.tsx (saves to localStorage, navigates to /edit).

Session Playback: EditPage (mounts) -> useLocalStorage (loads RecordingSession) -> <video> onTimeUpdate (sets currentTime) -> useSessionPlayback(session, currentTime) -> findStateAtTime (calculates state for that millisecond) -> EditPage (receives playbackState and renders overlays).

function findStateAtTime<T>(track: ComponentTrack<T> | undefined, timeMs: number): T | undefined {
if (!track || track.keyframes.length === 0) return undefined;
let foundKeyframe: Keyframe<T> | undefined;
for (let i = 0; i < track.keyframes.length; i++) {
const keyframe = track.keyframes[i];
if (keyframe.timestamp <= timeMs) {
foundKeyframe = keyframe;
} else {
break; // Keyframes are sorted, so we can stop.
}
}
return foundKeyframe ? foundKeyframe.state : track.keyframes[0]?.state;
}
Logic: This is the heart of the editor. It efficiently finds the correct state for any given millisecond by iterating through a sorted keyframe track and stopping as soon as it passes the target time.

Untested Areas: The entire application is untested from a unit/integration perspective. Logic in useSessionPlayback, useRecordingSession, and lib/ai.ts is complex and would benefit greatly from testing.

17. Potential Problem Areas
    ⚠️ Deprecated API: TextEditingToolbar.tsx relies heavily on document.execCommand for rich text formatting (bold, italic, lists, etc.). This is a deprecated, unreliable API that is a significant cross-browser reliability risk.

⚠️ Massive State Object: The entire application state for all scenes is stored in a single scenes array in Index.tsx and managed with useState. Modifying any single overlay requires re-rendering the entire VideoCanvas and passing new props down. This could cause severe performance bottlenecks as the number of scenes and overlays grows.

⚠️ Prop Drilling: Index.tsx passes over 60 props to VideoCanvas.tsx. This is extremely difficult to maintain, debug, and reason about.

🐛 Confusing State Flow: Index.tsx defines a sidebarProps object that it passes to FloatingControlsPanel. It also passes many of those same properties individually to VideoCanvas, which also has a sidebarProps prop with a different shape. This is redundant and confusing.

⚠️ Keyframe Spam: Index.tsx's useEffect hook creates a setInterval that calls takeSnapshot every 250ms. takeSnapshot records a new keyframe for every single active overlay, caption, and layout. If nothing has changed, this still generates thousands of redundant keyframes, bloating the RecordingSession object.

⚠️ Global Listeners: Index.tsx adds multiple global window event listeners (drop, paste, keydown). While they have cleanup, this can be fragile, especially in a multi-scene app.

⚠️ Inconsistent Drag/Resize Logic:

DraggableBrowser, DraggableFileViewer, DraggableTextOverlay, ExcalidrawOverlay, and VideoCanvas (for captions) use react-rnd.

DraggableGraph uses a completely custom onMouseDown / onMouseMove logic. This inconsistency makes maintenance harder.
